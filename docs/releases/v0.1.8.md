Changes since v0.1.7.

A security-hardening release: follow-up fixes from a full design + implementation review.
No schema, wire-format, or API changes - signatures stay byte-compatible and convergence is
unchanged. All fixes carry regression tests.

## Highlights

### Viewer console: XSS closed
Entity and type names are untrusted (they come from `observe`, including federation sync -
Principle 18), and the viewer interpolated them into the console's HTML. Two sinks are now closed:

- `esc()` escapes quotes (`"` and `'`) in addition to `< & >`, closing attribute-injection XSS
  in every `title="..."` / `data-id="..."` sink (a name like `x" onmouseover=...` could otherwise
  break out of the attribute into an event handler).
- The node hover tooltip now routes the node name/type/trust through `esc()` instead of raw
  interpolation, closing the stored-XSS vector (a malicious entity name executing on hover).

### HTTP surfaces: DNS-rebinding and CSRF defenses
Both loopback HTTP surfaces (the viewer and the MCP streamable-http daemon) now defend against a
browser being tricked into driving them from a foreign page.

- DNS rebinding: a loopback request whose `Host` is not a loopback name is refused - on every
  viewer path (reads and SSE included), and on the MCP daemon via a middleware that also checks
  `Origin` (the MCP-spec defense). A rebound page (`attacker.com` -> `127.0.0.1`) always carries a
  foreign hostname, so it is turned away.
- CSRF on the verdict endpoint: `/api/review` (which can mutate the canon via `entity_merge`) now
  requires the viewer's custom `X-Supragnosis-Viz` request header. A cross-site `<img>`/`<form>`
  cannot set a custom header, and a cross-site `fetch` that tries is blocked by a CORS preflight the
  server never approves; a same-origin fetch from the viewer sends it freely. This holds on every
  browser (the previous `Sec-Fetch-Site`-only check failed open on pre-2023 browsers).

### Sync ingest: signed-but-malformed peer events refused
A signature proves who sent an event, not that its content holds (Principle 18). The sync apply
path now runs the same well-formedness and confidence-range gate the local `observe` path enforces
(Principles 1/2), so a signed peer event carrying an out-of-range confidence or an empty-referent
assertion is rejected before it reaches the permanent append-only log. Per-event rejection is
convergence-safe (federation.md F7), and the rejection is reported, never silent (Principle 5).

### Store: a corrupt row can no longer erase itself
`observation_from_row` treated an unparseable `assertions`/`derived_from` value as empty, so a
routine re-arrival at the same content address would rewrite the row with those fields erased -
a destructive overwrite (Principle 3), and `assertions` are part of the content id and the
reprojection input. A present-but-unparseable value is now surfaced as a backend error instead of a
silent empty, so the re-arrival refuses rather than overwrites. An absent key is still a legitimate
empty value, and the embedding stays lenient (a recall aid, Principle 19).

### Signing bytes: the HLC is exhaustively destructured (Principle 14)
`attestation_signing_bytes` accessed the nested HLC by field, so adding an HLC field would have
compiled silently and let that field ride the wire while staying outside the signed bytes - a field
a relay could forge unnoticed. It is now destructured exhaustively, forcing an explicit
signed-versus-derived decision at compile time. The emitted bytes are unchanged, so existing
signatures still verify.
