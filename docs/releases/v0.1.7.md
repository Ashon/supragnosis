Changes since v0.1.6.

## Highlights

### tbox_change proposals get a belief-diff preview
Clicking a proposal used to preview the change on the graph only for `entity_merge` (the fold
arrow to the canonical node). A `tbox_change` - which redefines the T-Box, not a node - lit up
nothing: it has no `into` node, and its target is an observation, not a graph node. Proposals now
carry a structured `affected_types` list (`{target, name}`), and selecting one highlights every
edge whose relation kind, and every node whose entity type, the change (re)defines - framing the
affected members and listing the scope as chips in the Proposals panel.

- `propose` (MCP) accepts `affected_types`, each `{target: "entity" | "relation", name}`. Relation
  names are normalized to the graph's edge kinds so the highlight matches exactly.
- Additive and compatible: the field rides in the proposal's opened-event payload, folds to empty
  for proposals opened before it existed, and is ignored by older nodes - no storage/wire change.

### Server-mode peer markers
A hub gathers many peers, and having the camera jump to every remote sync hit was disorienting. On
a hub (auto-detected from the federation role) the viewer stops chasing hits and instead shows each
federated peer as a minimal cursor-dot - colored per peer, with a hover tooltip (id / last action /
hits). A new `[peers]` toolbar toggle turns it on/off; loopback/client viewers keep the existing
follow behavior.

- **Node/edge tour**: on an upload the peer's marker tours exactly what it contributed - hopping
  node -> edge -> node, ringing each node and flashing each edge with a brief dwell - derived by
  diffing the graph across the post-sync reload. Pulls and heartbeats carry no attributable nodes,
  so they stay a soft glide toward the active area (no fabricated node claims).
- The roster is seeded from `/api/federation` and refreshed, so idle peers appear as dim dots
  before their next live hit.

### Fix: the graph stays on screen during updates
The layout loader used to cover the whole graph on every incremental add/remove (an agent
observing, a peer pushing), not just the first load - which hid the very peer-marker tour a push
triggers. Only the first data load now settles behind the loader; later updates warm the layout
simulation below the loader threshold, so the graph stays visible and new nodes ease in.

### Fix: sync build warning
Dropped a dead binding in the sync advertise handler - the workspace is warning-clean.

## Install

```sh
curl -fsSL https://raw.githubusercontent.com/Ashon/supragnosis/main/scripts/install.sh | sh
```

Or download the platform `tar.gz` from the assets below, extract it, and put `supragnosis`
on your PATH. Supported: macOS (arm64 / x86_64), Linux (x86_64, glibc 2.35+); each asset ships a
`.sha256`.

- CLI: `supragnosis --help`. Register with an MCP client: `claude mcp add supragnosis -- ~/.local/bin/supragnosis`.

## Notes / limitations

- **Additive proposal field**: `affected_types` is optional and backward/forward compatible - v0.1.6
  nodes and hubs interoperate unchanged (storage and wire formats are unchanged).
- The peer-marker node/edge tour is driven by what the viewer can attribute: an upload (push) tours
  the exact added nodes/edges; pulls, heartbeats, and federated-search reads do not carry per-node
  detail to the viewer yet, so they render as a soft glide. Richer explore-side attribution would
  need a `nodes` field on the sync activity event (a follow-up).
- The viewer's opt-in network exposure (`SUPRAGNOSIS_VIZ_PUBLIC=1`) remains read-only and broader
  than the sync share whitelist; `/api/review` stays loopback-gated, so a remote browser cannot cast
  a verdict - the accept/reject gate is the loopback surface (or the future authenticated read tier,
  federation Phase 3.5). See architecture.md.
- **0.x early release** - the storage format and surfaces may still change.
